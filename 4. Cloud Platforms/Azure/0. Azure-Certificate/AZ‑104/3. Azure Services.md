# Azure Cloud Shell

## What is Azure Cloud Shell?
Azure Cloud Shell is a browser-based command-line environment that allows you to manage Azure resources directly from the Azure portal or a web browser without installing any tools locally.

## How does Azure Cloud Shell work?
Azure Cloud Shell runs in a temporary Linux or Windows-based container and provides authenticated access to Azure using your Azure account. Your scripts and files are stored in an Azure File Share and persist across sessions.

## Access Azure Cloud Shell
- From Azure Portal  
- Direct link: https://shell.azure.com  

## Available Shell Environments
- **PowerShell** â€“ Preferred by Windows users  
- **Bash (CLI)** â€“ Preferred by Linux users

- Persistent storage backed by Azure File Share

## When Should You Use Azure Cloud Shell?

As an IT Admin for Contoso Corporation, Azure Cloud Shell is useful when you need a quick and secure way to manage Azure resources from the command line without relying on your default administrative device.

### You can use Azure Cloud Shell to:
- Open a secure command-line session from any browser-based device  
- Interact with Azure resources without installing plug-ins or add-ons  
- Persist files between sessions for later use  
- Use either Bash or PowerShell to manage Azure resources  
- Edit files (such as scripts) using the Cloud Shell editor  

### You shouldnâ€™t use Azure Cloud Shell if:
- You plan to run long-running scripts or keep a session open for more than 20 minutes, as the session may disconnect without warning  
- You require admin-level permissions (such as `sudo`) within Azure CLI or PowerShell  
- You need to install unsupported tools that require a custom VM or container  
- You require storage across multiple regions, as Cloud Shell supports storage in only one region  
- You need multiple concurrent sessions, since Cloud Shell allows only one active instance at a time

### Introduction
Azure Resource Manager (ARM) templates enable you to implement **infrastructure as code (IaC)**. In this unit, you explore the structure of ARM templates, learn how to deploy them to Azure, and understand the purpose of the **resources** section.

---

## What is Infrastructure as Code (IaC)?

Infrastructure as code allows you to define and manage the infrastructure required for your application using code.

### Benefits of Infrastructure as Code
- Consistent configurations  
- Improved scalability  
- Faster deployments  
- Better traceability  

---

## What is an ARM Template?

An **ARM template** is a JSON file that defines the infrastructure and configuration for Azure resources. It enables declarative deployment, meaning you describe *what* you want to deploy, and Azure handles *how* to deploy it.

---

## Benefits of Using ARM Templates

- Declarative syntax  
- Repeatable and consistent deployments  
- Version-controlled infrastructure  
- Integrated dependency management  
- Native support within Azure  

---

## ARM Template File Structure

ARM templates consist of the following elements:

| Element | Description |
|------|------------|
| **$schema** | Required. Defines the JSON schema location and structure for the template. The version depends on deployment scope and editor support. |
| **contentVersion** | Required. Specifies the version of the template (for example, `1.0.0.0`) to track significant changes. |
| **apiProfile** | Optional. Defines a collection of API versions for resource types to avoid specifying API versions per resource. |
| **parameters** | Optional. Defines values provided during deployment via parameter files, command line, or Azure portal. |
| **variables** | Optional. Stores values to simplify expressions within the template. |
| **functions** | Optional. Defines user-defined functions for reuse of complex expressions. |
| **resources** | Required. Specifies the Azure resources to deploy or update. |
| **outputs** | Optional. Returns values at the end of deployment. |

---

## Deploy an ARM Template to Azure

You can deploy ARM templates using the following methods:

- Deploy a **local template**  
- Deploy a **linked template**  
- Deploy through a **continuous deployment (CI/CD) pipeline**
