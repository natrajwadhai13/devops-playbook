<a id="1. Error: CrashLoopBackOff"></a>

## 1. Error: CrashLoopBackOff

**Description:**

A pod’s container keeps crashing again and again. Kubernetes tries to restart it, but every attempt fails. This repeated crash–restart cycle is called CrashLoopBackOff.

**Solution:**

## 📌 Common Causes

- **Application bug** → The code itself is failing.
- **Missing environment variables** → ConfigMap or Secret not injected properly.
- **Wrong command/entrypoint** → Dockerfile CMD/args misconfigured.
- **Resource limits exceeded** → Pod runs out of memory or CPU.
- **Dependency failure** → Database or service not reachable, causing the app to exit.

## 🔍 How to Debug

1. **Check pod status**
   ```bash
   kubectl get pods
   ```
2. **Describe pod for events**
   ```bash
   kubectl describe pod <pod-name>
   ```
   → Look at the “Events” section for crash reasons.
3. **Check container logs**
   ```bash
   kubectl logs <pod-name> --previous
   ```
   → See the last crash output.
4. **Check resource usage**
   ```bash
   kubectl top pod <pod-name>
   ```
   → Verify CPU/memory consumption.

## ✅ Fix Strategies

- **Fix application bug** → Correct the code error.
- **Add missing configs** → Ensure environment variables, ConfigMaps, and Secrets are set.
- **Adjust resources** → Increase memory/CPU limits in the pod spec.
- **Use startupProbe** → If the app needs extra time to start, configure a startup probe.
- **Verify image** → Check Docker image CMD/ENTRYPOINT correctness.

👉 In interviews, the best way to answer is:

**“CrashLoopBackOff occurs when a container keeps crashing. I would debug it using `kubectl describe` and `kubectl logs`, check for missing configs or resource limits, and then fix by correcting the application or adjusting the pod specification.”**

<a id="2. Error: ImagePullBackOff"></a>

## 2. Error: ImagePullBackOff

**Description:**  
Kubernetes is unable to pull the specified container image from the registry.

The pod cannot pull the container image from the registry. Kubernetes keeps retrying, but the image download fails, so the pod goes into ImagePullBackOff state.

**Solution:**

## 📌 Common Causes

- **Wrong image name or tag** → Typo in image name or using a non‑existent tag.
- **Registry authentication failure** → Missing or incorrect `imagePullSecrets`.
- **Private registry access denied** → No credentials provided for private repo.
- **Network/DNS issues** → Cluster cannot reach Docker Hub or registry.
- **Rate limits** → Registry (like Docker Hub) blocking too many requests.

## 🔍 How to Debug

1. **Check pod events**
   ```bash
   kubectl describe pod <pod-name>
   ```
   → Look for “Failed to pull image” messages.
2. **Verify image name/tag**
   ```bash
   docker pull <image-name>:<tag>
   ```
   → Test locally to confirm image exists.
3. **Check secrets**
   ```bash
   kubectl get secrets
   ```
   → Ensure `imagePullSecrets` are configured.
4. **Check registry access**
   - Ping registry URL.
   - Verify credentials.

## ✅ Fix Strategies

- **Correct image name/tag** → Use valid image and tag.
- **Add imagePullSecrets** → Create secret with registry credentials:
  ```bash
  kubectl create secret docker-registry myregistrykey \
    --docker-server=<registry-url> \
    --docker-username=<user> \
    --docker-password=<password> \
    --docker-email=<email>
  ```
  Then reference it in pod spec.
- **Use proper registry URL** → For private registries, specify full path.
- **Handle rate limits** → Use authenticated pulls or mirror registry.
- **Check network** → Ensure nodes can reach registry.

👉 In interviews, a strong answer is:

**“ImagePullBackOff occurs when Kubernetes cannot pull the container image. I would debug using `kubectl describe pod`, check the image name and tag, verify registry credentials with `imagePullSecrets`, and fix by correcting the image reference or adding proper authentication.”**

<a id="3. Error: Unauthorized"></a>

## 3. Error: Unauthorized

**Description:**  
This error occurs when the user or service account does not have permission to perform the requested operation.

This error means the request was rejected because the user, pod, or service account does not have the required permissions. In Kubernetes, this usually happens due to RBAC (Role‑Based Access Control) misconfiguration or missing credentials.

**Solution:**

## 📌 Common Causes

- RBAC misconfiguration → Role/ClusterRole not bound correctly.
- ServiceAccount lacks permissions.
- Invalid/expired token in kubeconfig.
- Trying to access resources outside allowed scope.
- Namespace mismatch between role and pod.

## 🔍 How to Debug

1. **Check pod events**
   ```bash
   kubectl describe pod <pod-name>
   ```
   → Look for “Unauthorized” messages.
2. **Verify ServiceAccount**
   ```bash
   kubectl get serviceaccount <sa-name> -n <namespace>
   ```
3. **Check role bindings**
   ```bash
   kubectl get rolebinding,clusterrolebinding -n <namespace>
   ```
4. **Inspect kubeconfig** → Ensure correct user context and token.

## ✅ Fix Strategies

- Bind the correct Role/ClusterRole to the ServiceAccount.
  ```bash
  kubectl create rolebinding myapp-rb \
    --role=myapp-role \
    --serviceaccount=default:myapp-sa \
    --namespace=myapp-ns
  ```
- Update kubeconfig with valid credentials.
- Add missing verbs (get, list, watch, create, delete) in RBAC rules.
- Ensure roles and bindings are in the same namespace.

👉 **Interview Answer Template:**

“Unauthorized occurs when a pod or user doesn’t have permission to access a resource. I would debug using `kubectl describe` and check service accounts and role bindings. The fix is usually to correct RBAC configuration, bind the right role to the service account, or update credentials.”

<a id="4. Error: DeadlineExceeded"></a>

## 4. Error: DeadlineExceeded

**Description:**  
This error indicates that the requested operation did not complete within the specified timeout period.

This error means a pod, job, or request has run longer than the allowed time and Kubernetes terminated it. Essentially, the operation exceeded its deadline or timeout.

**Solution:**

- Increase the timeout values for the operation if possible, either by adjusting Kubernetes configurations or by retrying the operation with a longer timeout.

- Optimize the operation to reduce the time it takes to complete. This might involve improving the efficiency of the application code or optimizing resource usage.

- Scale resources such as nodes or pods to distribute the workload and reduce the likelihood of hitting timeout limits.

## 📌 Common Causes

- **Job timeout** → A Kubernetes Job exceeded its `activeDeadlineSeconds`.
- **Pod startup delay** → Container took too long to start, failing readiness/startup probes.
- **API request timeout** → Long‑running API call exceeded the client/server timeout.
- **Resource bottleneck** → Pod stuck waiting for CPU/memory, causing delays.
- **Network issues** → Slow or unreachable external service caused request timeout.

## 🔍 How to Debug

1. **Check pod/job events**
   ```bash
   kubectl describe pod <pod-name>
   kubectl describe job <job-name>
   ```
   → Look for “DeadlineExceeded” in events.
2. **Check logs**
   ```bash
   kubectl logs <pod-name>
   ```
   → See if the app was stuck or waiting.
3. **Inspect probes** → Verify readiness/liveness/startup probe settings.
4. **Check resource usage**
   ```bash
   kubectl top pod <pod-name>
   ```
   → Confirm if CPU/memory shortage caused delays.

## ✅ Fix Strategies

- **Increase deadline** → Adjust `activeDeadlineSeconds` for jobs or probes.
- **Optimize application** → Reduce startup time or long‑running tasks.
- **Tune probes** → Configure realistic `timeoutSeconds` and `periodSeconds`.
- **Add resources** → Increase CPU/memory requests and limits.
- **Check dependencies** → Ensure external services respond within expected time.

👉 **Interview Answer Template:**  
“DeadlineExceeded occurs when a pod, job, or request runs longer than its allowed time. I would debug using `kubectl describe` and logs, check probe settings and resource usage, and fix by increasing deadlines, tuning probes, or optimizing the application and dependencies.”

<a id="5. Error: Invalid YAML"></a>

## 5. Error: Invalid YAML

**Description:**  
Kubernetes configuration files contain syntax errors, making them invalid and unable to be applied.

This error means the YAML manifest you applied has a syntax or formatting problem. Kubernetes cannot parse the file correctly, so the resource fails to create.

**Solution:**

- Validate the YAML syntax using tools like `kubectl apply --dry-run` or YAML linting tools before applying them to the cluster.

- Double-check the structure, indentation, and syntax of the YAML files to ensure they comply with Kubernetes specifications.

- Use YAML editors or IDE plugins that provide syntax highlighting and error checking to catch issues before applying configurations.

## 📌 Common Causes

- Wrong indentation (extra/missing spaces).
- Tabs used instead of spaces.
- Missing `:` after a key.
- Wrong data type (string vs integer vs boolean).
- Improper nesting of keys.
- Invalid field names not supported by Kubernetes API.

## 🔍 How to Debug

1. **Validate YAML syntax**
   ```bash
   yamllint <file>.yaml
   ```
2. **Dry‑run apply**
   ```bash
   kubectl apply --dry-run=client -f <file>.yaml
   ```
   → Shows parsing errors without creating resources.
3. **Check indentation carefully** → YAML is space‑sensitive.
4. **Compare with official docs** → Ensure field names are valid for that resource.

## ✅ Fix Strategies

- Use **spaces only** (no tabs).
- Keep consistent indentation (usually 2 spaces per level).
- Add missing `:` after keys.
- Correct data types (quotes for strings, no quotes for numbers/booleans).
- Validate with `yamllint` or VS Code YAML extension.

- Adjust pod resource requests/limits to fit node capacity.
- Use **PodPriority** carefully to avoid unnecessary evictions.

## 68. Error: Insufficient Memory

**Description:**  
Pods fail to be scheduled due to insufficient memory resources on the nodes.

**Solution:**

- Increase the memory resources available on the nodes in the cluster. This can be done by adding more nodes or increasing the memory allocation for existing nodes.

- Review the resource requests and limits specified in the pod's YAML file. Adjust them to ensure they match the available resources on the nodes and the requirements of the application.

- Optimize the application's memory usage by identifying memory leaks or inefficiencies in the code.

## 15. Error: PodTerminating

**Description:**  
Pods are in the terminating state but fail to be deleted completely.

**Solution:**

- Check the events and logs for the pod to identify any errors or issues preventing termination.
- Ensure that the pod's associated resources, such as PersistentVolumeClaims or ConfigMaps, are released properly.
- If the pod remains stuck in terminating state, force delete it using the kubectl delete pod `<pod_name>` --grace-period=0 --force command.

- Add additional storage capacity to the node if possible, either by attaching more disks or expanding existing volumes.
