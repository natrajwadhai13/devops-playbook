<a id="q1-high-availability"></a>

### **Q1. Scenario: Your firm is experiencing frequent downtimes. How would you leverage Kubernetes to achieve high availability for your applications? (Achieving High Availability in Kubernetes)**

### Answer

- In Kubernetes, achieving high availability involves several practices. Applications are deployed using **Deployments or StatefulSets** depending on whether they are stateless or stateful.

- Deployments allow scaling replicas so if one pod fails, others continue serving traffic. StatefulSets help manage stateful workloads with persistent identity and storage.

- A **Kubernetes Service** distributes traffic across pods using load balancing. If a pod fails, traffic is automatically redirected to healthy pods.

- Health checks such as **liveness probes and readiness probes** allow Kubernetes to detect unhealthy containers and restart them automatically.

High availability can also be improved by:

- Running nodes across multiple zones
- Using **Horizontal Pod Autoscaling (HPA)**
- Automatically scaling pods based on CPU or memory load

### Layman Language

- Imagine multiple toy robots doing the same task. If one robot stops working, others continue doing the work.

- Kubernetes ensures multiple copies of your application run at the same time. If one fails, another continues.

- It also spreads applications across different locations and adds more copies automatically when workload increases.

<a id="q2-zero-downtime"></a>

### **Q2. Scenario: How can your company ensure zero downtime deployments with Kubernetes? What strategies and tools would you use to accomplish this ? (Ensuring Zero Downtime Deployments)**

### Answer

- Zero downtime deployments in Kubernetes can be achieved using the **RollingUpdate strategy**.

- RollingUpdate gradually replaces old application versions with new ones while keeping some instances running.

Deployment parameters such as:

- **maxUnavailable**
- **maxSurge**

control how many pods can be unavailable during the update.

Setting:

```
maxUnavailable: 0
```

ensures no pods go down during deployment.

Readiness probes ensure new pods receive traffic only when fully ready.

Tools such as:

- **Helm**
- **ArgoCD**

- can help manage versioned releases and enable easy rollback if issues occur.

### Layman Language

- Imagine replacing old cash registers in a store one at a time so the store never closes.

- Kubernetes updates applications one instance at a time so customers always have a working system.

<a id="q11-cluster-scaling"></a>

# Q11. Scenario: Your Kubernetes cluster is reaching resource limits during peak traffic. How would you scale the cluster to handle increased demand?

## Answer

To handle increased demand in Kubernetes, I would implement **cluster autoscaling and pod autoscaling**.

First, I would enable the **Horizontal Pod Autoscaler (HPA)** to automatically increase the number of pod replicas based on CPU, memory, or custom metrics.

Next, I would configure the **Cluster Autoscaler**, which automatically adds or removes nodes in the Kubernetes cluster depending on pod scheduling requirements. If pods cannot be scheduled due to insufficient resources, the autoscaler provisions new nodes.

Additionally, I would ensure that resource requests and limits are properly configured for containers to allow Kubernetes to make accurate scaling decisions.

For traffic management, I would use **load balancers and ingress controllers** to distribute traffic evenly across the pods.

Monitoring tools such as **Prometheus and Grafana** would help identify scaling trends and ensure the cluster scales efficiently.

## Layman Language

Imagine your restaurant suddenly becomes very busy. Instead of making customers wait, you quickly add more tables and hire extra staff.

Kubernetes does the same thing. When more users arrive, it automatically adds more application copies and even new servers if needed.

<a id="q12-pod-crash"></a>

# Q12. Scenario: Your application pods are crashing frequently in production. How would you troubleshoot the issue?

## Answer

To troubleshoot crashing pods, I would start by checking pod logs using:

```
kubectl logs <pod-name>
```

Next, I would inspect the pod status and events using:

```
kubectl describe pod <pod-name>
```

This helps identify errors such as image issues, configuration problems, or resource constraints.

I would also check resource usage using:

```
kubectl top pod
```

to determine if the pod is exceeding its CPU or memory limits.

If the issue persists, I would review container configurations, environment variables, secrets, and dependencies.

Monitoring tools like **Prometheus**, logging tools like **ELK Stack**, and tracing tools like **Jaeger** can provide deeper insights into application failures.

## Layman Language

If a machine in a factory stops working, the first step is checking its error messages and logs to understand what went wrong.

Kubernetes troubleshooting works the same way — you check logs, system messages, and resource usage to find the cause.

<a id="q13-networking-issue"></a>

# Q13. Scenario: Pods in your Kubernetes cluster cannot communicate with each other. How would you diagnose and resolve this issue?

## Answer

Pod communication issues can be caused by several factors.

First, I would verify pod connectivity using commands such as:

```
kubectl exec -it <pod-name> -- ping <other-pod-ip>
```

Next, I would check **Kubernetes Services** to ensure the service configuration correctly maps to the target pods.

I would also inspect **Network Policies** to ensure they are not blocking communication between pods.

If using a CNI plugin like **Calico**, **Flannel**, or **Cilium**, I would verify that the networking components are functioning correctly.

Finally, I would examine DNS resolution using:

```
nslookup service-name
```

to ensure services resolve correctly inside the cluster.

## Layman Language

Imagine offices in a building that cannot call each other because the phone lines are blocked.

The solution is to check the wiring, communication rules, and phone system configuration.

<a id="q21-node-failure"></a>

# Q21. Scenario: One of the worker nodes in your Kubernetes cluster suddenly fails. How would Kubernetes handle this situation and what actions would you take?

## Answer

When a worker node fails, the Kubernetes control plane detects the node failure through the **node controller**.

Pods running on that node become **unavailable**. If those pods are managed by controllers like **Deployments, ReplicaSets, or StatefulSets**, Kubernetes automatically schedules replacement pods on healthy nodes.

To ensure faster recovery, I would:

- Configure proper **replica counts**
- Use **PodDisruptionBudgets**
- Enable **Cluster Autoscaler** to add nodes if capacity is insufficient
- Monitor node health using tools like **Prometheus and Grafana**

If the node failure is due to infrastructure issues, I would also investigate logs, cloud provider events, and node metrics.

## Layman Language

Imagine a delivery truck breaks down. The system quickly assigns other trucks to deliver the packages so customers still receive their orders.

<a id="q22-k8s-upgrade"></a>

# Q22. Scenario: How would you upgrade a Kubernetes cluster in production without causing downtime?

## Answer

Upgrading Kubernetes clusters requires careful planning.

Steps include:

1. Upgrade the **control plane components**
2. Upgrade **worker nodes gradually**
3. Drain nodes before upgrading:

```
kubectl drain <node-name>
```

This safely evicts pods and reschedules them on other nodes.

4. Upgrade kubelet and node components
5. Uncordon the node:

```
kubectl uncordon <node-name>
```

Using **rolling upgrades** ensures minimal service disruption.

Cluster upgrade strategies also involve testing upgrades in staging environments first.

## Layman Language

It’s like upgrading machines in a factory one at a time so production never stops.

Container logs are stored on the node filesystem.

Logs can be accessed using:

```
kubectl logs <pod-name>
```

Centralized logging solutions include:

- ELK Stack
- Loki
- Fluentd

These tools collect logs across all nodes and pods.

## Layman Language

It’s like collecting reports from every machine in a factory into one central system.

<a id="q35-pod-affinity"></a>

# Q35. Scenario: What are pod affinity and anti-affinity?

## Answer

Pod affinity controls where pods should be scheduled relative to other pods.

Pod anti-affinity prevents pods from running on the same node.

These rules help improve:

- availability
- workload distribution

## Layman Language

It’s like deciding which employees should work together and which should stay apart.
